Security and Compliance (امنیت و انطباق)

این سند، استانداردها، سیاست‌ها و رویه‌های امنیتی و انطباق (Compliance) پلتفرم Vira AI را برای تضمین محافظت از داده‌های کاربران، مدل‌ها و زیرساخت، تشریح می‌کند.

۱. امنیت احراز هویت و دسترسی (Authentication and Access Security)

۱.۱. مدیریت کلید API (API Key Management)

تولید: کلیدهای API باید با استفاده از الگوریتم‌های رمزنگاری قوی و غیرقابل حدس تولید شوند و دارای طول حداقل ۳۲ کاراکتر باشند.

ذخیره‌سازی: کلیدها نباید به صورت متن ساده (Plain Text) ذخیره شوند؛ بلکه باید از هش‌سازی یک‌طرفه (One-way Hashing) یا سرویس‌های مدیریت کلید (Key Management Services - KMS) استفاده شود.

چرخه حیات: کلیدها باید قابلیت لغو (Revocation) و تمدید (Rotation) دوره‌ای داشته باشند (توصیه: هر ۹۰ روز یکبار).

۱.۲. کنترل دسترسی مبتنی بر نقش (Role-Based Access Control - RBAC)

دسترسی به منابع مختلف (مانند مدل‌ها، مجموعه‌های داده و گزارش‌ها) باید بر اساس نقش کاربر تعیین شود. نقش‌های اصلی شامل موارد زیر است:

Admin: دسترسی کامل به تمام سیستم‌ها و تنظیمات.

Developer: دسترسی به آپلود کد، ساخت مدل و اجرای پیش‌بینی.

Viewer: دسترسی فقط خواندنی (Read-Only) به گزارش‌ها و نتایج پیش‌بینی.

۲. حفاظت از داده (Data Protection)

۲.۱. رمزنگاری در حین انتقال (Encryption in Transit)

تمام ارتباطات بین کاربران و API پلتفرم Vira AI باید با استفاده از پروتکل‌های امن و به‌روز مانند TLS 1.2 یا بالاتر رمزنگاری شوند. (استفاده از HTTP ممنوع است).

۲.۲. رمزنگاری در حالت سکون (Encryption at Rest)

تمام داده‌های ذخیره‌شده (مجموعه‌های داده، پارامترهای مدل، نتایج پیش‌بینی) باید در بستر ذخیره‌سازی، رمزنگاری شوند.

قانون: تمام دیسک‌های ذخیره‌سازی (مانند پایگاه‌های داده و Object Storage) باید با استفاده از رمزنگاری AES-256 رمزگذاری شوند.

۲.۳. حذف و نگهداری داده (Data Retention and Deletion)

سیاست نگهداری: سیاست‌های مشخصی برای نگهداری داده‌ها تعریف شده و کاربران می‌توانند داده‌های خود را در هر زمانی حذف کنند.

حذف ایمن: هنگام حذف داده‌ها، باید از روش‌هایی استفاده شود که بازیابی اطلاعات را غیرممکن کند.

۳. مدیریت آسیب‌پذیری و نظارت (Vulnerability Management and Monitoring)

۳.۱. تست نفوذ و ارزیابی آسیب‌پذیری

دوره‌ای: ارزیابی‌های آسیب‌پذیری و تست‌های نفوذ (Penetration Testing) باید حداقل سالیانه توسط یک تیم مستقل انجام شود.

پچ‌گذاری: به‌روزرسانی‌های امنیتی (Patching) برای تمام سیستم‌عامل‌ها، کتابخانه‌ها و فریم‌ورک‌های مورد استفاده باید به صورت منظم اعمال شوند.

۳.۲. ثبت وقایع و نظارت (Logging and Monitoring)

تمام فعالیت‌های حساس در پلتفرم (مانند ورود به سیستم، اجرای مدل، تغییرات سطح دسترسی) باید ثبت (Logged) شوند.

سیستم‌های نظارت (Monitoring) باید برای شناسایی و هشداردهی سریع در مورد تلاش‌های دسترسی غیرمجاز یا ناهنجاری‌های سیستمی فعال باشند.
